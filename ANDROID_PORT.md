# ANDROID_PORT.md — план портирования Formula Synth на Android (Kotlin)

План «для себя»: что переносим, в каком порядке, где грабли. Исходник —
этот репозиторий (~6200 строк TS): чистое DSP-ядро + Web Audio FX-граф +
vanilla-DOM UI. Целевой вид — нативное приложение на Kotlin + Jetpack
Compose, отдельный репозиторий (`formula-synth-android`).

## 0. Ключевое архитектурное решение

**Нативный порт, не WebView-обёртка.** Причины:

- WebView/TWA дал бы «порт за день», но: latency и глюки Web Audio в
  WebView, нет контроля над аудиосессией/фокусом, фоновое воспроизведение
  через костыли, и это не «Android (Kotlin)», а тот же веб.
- Синтезатор генеративный (не клавиатурный) — жёсткая latency не нужна,
  поэтому **AudioTrack (float PCM) из Kotlin достаточно**, JNI/Oboe/C++ не
  тащим. План Б, если упрёмся в GC-джиттер на аудиопотоке: вынести только
  render-цикл в Oboe (архитектура ниже это позволяет — движок и так чистый).
- Огромный бонус нативного порта: **весь движок, включая FX, становится
  чистым Kotlin-кодом** → офлайн-рендер в юнит-тестах. В вебе FX жили в
  нативных нодах и проверялись только браузерным smoke/rec; тут вся
  цепочка тестируется vitest-стилем (JUnit), объективно, по семплам.

**Главная новая работа — FX.** В вебе Filter/Chorus/Phaser/Delay/Reverb/
Limiter — бесплатные нативные ноды (`createBiquadFilter`, `createConvolver`,
`createDynamicsCompressor`, `createDelay`, LFO — `OscillatorNode`). В Kotlin
всё это пишем сами (~800–1200 строк DSP + тесты). DSP-генераторы, наоборот,
переносятся почти механически.

**Стек:** Kotlin, Jetpack Compose (UI), AudioTrack float
(`AudioFormat.ENCODING_PCM_FLOAT`, minSdk 26), kotlinx.serialization (JSON
состояния), DataStore (пользовательские пресеты). Никаких аудио-библиотек —
дух «no runtime deps» сохраняем.

## 2. Порт DSP-ядра (dsp/)

Переносить посэмплово, сохраняя **Double** внутри (JS Number = double;
воркалет тоже считает в double и лишь пишет Float32 на выход). На выходе
блока — Float.

Грабли численной совместимости:

- **Битовые операции (bytebeat, mulberry32).** JS работает в int32 —
  Kotlin `Int` тоже int32 с тем же переполнением. Маппинг: `>>`→`shr`,
  `>>>`→`ushr`, `|`→`or`, `&`→`and`, `Math.imul`→обычный `*` на Int
  (одинаковый wraparound). Сдвиг на переменную величину: и JS, и Kotlin
  маскируют счётчик `&31` — совпадает. `x >>> 0` и деление на 2^32 в
  mulberry32 → `(x.toLong() and 0xFFFFFFFFL) / 4294967296.0`.
- **Транцендентные функции** (`sin`, `exp`, `pow`) НЕ гарантируют
  побитовое совпадение JS↔JVM/ART. Допуск golden 1e-9 может не пройти —
  стратегия: сначала гоняем с 1e-9, где расходится — ослабляем допуск до
  1e-6..1e-7 **точечно** и слушаем ушами (rossler/lorenz — хаос, там
  расхождение накапливается экспоненциально: для них сверять только
  первые N семплов либо свойства сигнала, не значения).
- `Float32Array` буферов (Karplus-Strong, фазы Шепарда) → `FloatArray`,
  сохранить именно float-точность там, где она была (golden фиксируют её).
- RNG инъецируется как в вебе: `mulberry32` в тестах, `Random` в проде.
  `Math.random` напрямую не звать — правило то же.

Golden-файлы (`tests/golden/formulas.json`, `mod.json`, сид 42, 4×128
семплов) **копируем как есть** в ресурсы androidTest/test — это и есть
контракт «звук совпадает с вебом».

## 3. FX-цепочка (fx/) — главная новая работа

Порядок графа как в engine.ts: `mix → Filter → Chorus → Phaser → Delay →
Reverb → Limiter → Master → (analyser tap)`. Каждый эффект — класс с
`process(block: FloatArray)` (или in/out), dry/wet и sum как в вебе.
Реализации:

- **Filter (мультирежимный).** Ноды Web Audio НЕ копируем 1-в-1 —
  достаточно следовать идее (формуле) ноды. 8 biquad-типов — коэффициенты
  по RBJ Audio EQ Cookbook, побитового совпадения с BiquadFilterNode не
  добиваемся. Formant = 3 параллельных bandpass на `vowelFormants`
  (Filters.kt уже чистый — гласные/частоты берём оттуда). Comb =
  delay-линия с обратной связью (~50 мс + gain fb).
- **Chorus/Flanger.** Модулируемая delay-линия (до 0.2 c) + LFO (в вебе —
  OscillatorNode sine) + feedback + dry/wet. LFO считаем сами в цикле.
- **Phaser.** Цепочка all-pass biquad-звеньев, число звеньев = Stages
  (реально пересобирается, как у нас). **Сохранить фикс свипа**: центр
  all-pass строго в [200 … 200+3600·Depth], LFO качает на полразмаха —
  иначе вернём баг щелчков, который ловили через `npm run rec`.
- **Delay.** Линия до 3 c + feedback + dry/wet. Тривиально.
- **Reverb.** В вебе — ConvolverNode с генерируемым импульсом (шумовой
  хвост с затуханием, пересчёт только при смене Decay). Варианты:
  (а) честная свёртка — партиционированная FFT-свёртка, ~день работы + свой
  FFT; (б) алгоритмический (FDN/Freeverb), звук ЗАМЕТНО другой. **Решение:
  (а)**, потому что философия проекта — «звук фиксирован golden'ами», и
  импульс мы генерируем сами тем же кодом → есть шанс на близкое
  совпадение. FFT свой (radix-2, ~100 строк), он же пригодится
  спектрограмме скоупа.
- **Limiter.** В вебе — DynamicsCompressorNode с настройками из engine.ts.
  Достаточно простого soft-knee лимитера по мотивам (feed-forward, атака/
  релиз как в настройках engine.ts) — задача узла страховочная, не
  звукоформирующая; на слух сверить, что пики держит так же.
- **Сглаживание параметров.** В вебе `setTargetAtTime(FX_SMOOTH_TC)` спасает
  от зиппера при FX-модуляции (щелчки S&H + петля фейзера). В Kotlin —
  однополюсный сглаживатель на каждый модулируемый параметр с той же
  постоянной времени. Это НЕ опция, это выученный урок.

Каждый эффект — свой unit-тест на импульсной/синусовой характеристике
(частота среза, время задержки, глубина хоруса и т.п.). Плюс общий
офлайн-рендер-тест всей цепочки: не тишина, конечен, не клиппит — аналог
audio-sanity.

## 4. Движок (engine/)

Вместо графа нод — **явный render-поток**:

- Поток с `THREAD_PRIORITY_URGENT_AUDIO`, цикл: собрать блок 128 семплов
  (размер блока = 128, как у воркалета — от него зависит block-rate
  модуляции и golden mod.json!), прогнать FX, `audioTrack.write(FloatArray,
  BLOCKING)`. Буфер AudioTrack — 2–4 блока сверх getMinBufferSize.
- **Ноль аллокаций в цикле** (иначе GC-щелчки): все буферы преаллоцированы,
  никаких лямбд/боксинга в горячем пути. Это дисциплина всего fx/dsp-кода.
- Gate как в вебе (dsp/gate.ts): фейд 5 блоков → выключенный генератор не
  считается и «замирает» (при включении продолжает с той же точки).
- Модуляция генераторов — block-rate, overwrite-then-restore, `modT` не
  сбрасывается в reset() — вся семантика из dsp/mod.ts переносится как
  есть. Разница с вебом: LFO и генераторы теперь в одном потоке, «синхронны
  без обмена сообщениями» получается само.
- Модуляция FX: в вебе — таймер ~40 Гц в main thread. Здесь проще и лучше:
  считать эффективный FxState **раз в блок прямо в render-цикле**
  (block-rate ≈ 375 Гц при 48 кГц) той же чистой `modulateFx`, со
  сглаживанием из п.3. Отдельный таймер не нужен; логика base/live
  (baseFx = слайдеры) сохраняется.
- Связь UI↔движок: UI пишет снапшот состояния (immutable data class) в
  `AtomicReference`, render-поток забирает в начале блока. Без локов в
  аудиопути. Команды типа «перестроить фильтр под новый filterType» —
  флаг в снапшоте (аналог applyFx vs applyFxParams).
- Analyser tap: кольцевой буфер последних N семплов (после лимитера) для
  скоупа; UI читает из него по своему таймеру.
- Recorder: те же RecorderCore/encodeWAV (порт), пишет пост-лимитер PCM,
  сохранение через MediaStore (Music/FormulaSynth) + share intent.

**Android-обвязка:**

- Foreground service c MediaSession (тип `mediaPlayback`) — синт играет со
  свёрнутым экраном, есть notification с Play/Stop. Это замена wakelock-у
  как «главному» механизму; FLAG_KEEP_SCREEN_ON — только пока скоуп открыт.
- Audio focus: `AUDIOFOCUS_GAIN`, при потере — **стоп** (решено), без
  дакинга; transient-потеря (звонок) — тоже стоп, проще и предсказуемее.
- iosUnlock не нужен вообще. Роскошь.

## 5. Состояние и пресеты

- **Share-токены `#s=…` и URL-параметры (`?preset=`) НЕ переносим** —
  решение пользователя. src/state/share.ts, диплинки, App Links — мимо.
  Кнопки Share в приложении нет.
- **JSON-формат состояния v3 сохраняем** (schema.ts →
  kotlinx.serialization с ручным терпимым разбором в духе sanitizeState:
  вся мусоростойкость переносится, порт тестов serialize.test.ts).
  Мотивация теперь не share-интероп, а пресеты: они приезжают из веб-репо
  в этом формате.
- **Встроенные пресеты и fxPresets — из веб-репо в виде JSON** (решено:
  пресеты годные, источник истины — веб). В вебе пишем маленький скрипт
  `scripts/dump-presets.mjs`, который дампит presets.ts + fxPresets.ts →
  JSON; Android кладёт его в assets и читает через тот же sanitize-разбор.
  Пресеты правятся в ОДНОМ месте, обновление в приложении = перекопировать
  файл. Тест на Android-стороне: каждый пресет из assets парсится без
  потерь (аналог presets.test.ts + audio-sanity по пресетам).
- Пользовательские пресеты: DataStore, JSON тот же. Ключи localStorage —
  контракт веба, тут не связывает; но имя хранилища выбрать один раз и
  заморозить сразу (выучили на formula_audio_lab_*).

## 6. UI (Compose)

Прямая проекция текущего UI (он уже mobile-first):

- Топбар: Play/Rec + панели-переключатели Scope/Effects/Mod (radio,
  максимум одна активна — setPanel), меню пресетов, Save (Share нет —
  токены не переносим). Master volume закреплён сверху.
- Карточки формул: LazyColumn (телефон) / LazyVerticalGrid в 2 колонки на
  планшете и в landscape (аналог грида ≥720px; порог — по
  `WindowSizeClass`, medium+). Планшетный лэйаут — требование первой
  версии, не «потом»: адаптивность закладывается сразу, а не
  ретрофитится. Свёрнуты по умолчанию: чек + название + формула;
  включение разворачивает; ↺ reset в шапке. Слайдеры — Compose Slider +
  свой +/− с автоповтором (порт adjust.ts) + индикатор ∿ на модулируемых.
- Панель Effects: карточки модулей, меню «Effects preset» сверху, контролы
  фильтра перестраиваются под filterType (порт updateFilterControls),
  ручная правка сбрасывает пресет в плейсхолдер.
- Панель Mod: порт modmatrix.ts — 4 LFO + таблица маршрутов; в Compose
  источник истины уже state, а не DOM — станет чище. Сохранить поведение:
  приёмники фильтруются до включённых формул, цель существующего роута
  видна как «(off)», группа Effects всегда есть.
- Scope: Compose Canvas, осциллограф + спектрограмма-водопад (FFT из п.3);
  рисование гасится при скрытой панели/фоне (аналог rAF-гейтинга).
- Формулы отображаем как текст (в вебе так же) — без MathML/LaTeX.

## 7. Тестирование

Стратегия ЛУЧШЕ вебовской, потому что движок весь чистый:

1. **Unit (JUnit, JVM, без эмулятора)** — порт всех vitest-тестов:
   golden (те же JSON, допуски по п.2), свойства сигналов, mod, gate,
   serialize, wav, presets, audio-sanity. Плюс новые тесты на каждый FX.
2. **Офлайн-рендер** (замена `npm run rec`): JVM-тест рендерит пресет
   N секунд через ПОЛНУЮ цепочку → метрики peak/RMS + детектор щелчков
   (порт из scripts/rec.mjs) + опционально WAV в build/ для ушей.
   Гоняется в CI на каждый коммит — в вебе такое требовало headless-Chromium.
3. **Instrumented smoke** (эмулятор, замена `npm run smoke`): каждый
   пресет грузится и играет 1 c, каждая формула включается, панели
   открываются — падение на любом крэше/ANR. Compose UI tests +
   скриншоты (замена `npm run shot`).
4. Ручное на устройстве: латентность/андерраны (systrace/Perfetto),
   фоновое воспроизведение, наушники/BT, поворот, doze.

TDD как в вебе: сначала тест, потом код; golden не перегенерировать
«чтобы прошло».

## 8. Этапы

- **M0 — скелет.** Пустой проект (Kotlin, Compose, minSdk 26), CI
  (JVM-тесты + lint + detekt), копия golden JSON в ресурсы. Правила линта
  в духе «никаких `as`-кастов» → detekt `UnsafeCast` и т.п.
- **M1 — DSP-ядро.** Порт rng/generator/gate/mod/wav/recorder + все их
  тесты. **Выход: golden зелёные** (с задокументированными допусками).
  Самый предсказуемый этап; сразу вскроет все числовые грабли.
- **M2 — звук.** AudioTrack-цикл, микс включённых генераторов, gate,
  master, Play/Stop с минимальным UI-огрызком. Выход: телефон играет fm.
- **M3 — модуляция генераторов.** Block-rate mod в цикле, golden mod.json.
- **M4 — FX.** По одному узлу с тестами: Filter (biquad→formant→comb) →
  Delay → Chorus → Phaser → Reverb (FFT + свёртка) → Limiter. Потом
  FX-модуляция + сглаживание. Выход: офлайн-рендер-тест всей цепочки,
  «Psychedelic melt (FX mod)» звучит как в вебе (сверка ушами + rec-WAV
  из веба рядом).
- **M5 — UI.** Compose-экраны по п.6, включая планшетный/landscape-лэйаут
  (WindowSizeClass, грид 2 колонки); скоуп последним (нужен FFT из M4).
  Instrumented-скриншоты гонять в обоих форм-факторах (phone + tablet).
- **M6 — состояние.** Схема/sanitize, дампер пресетов в веб-репо +
  assets-JSON + загрузчик, user-пресеты, Rec→WAV→MediaStore.
- **M7 — продакшн.** Foreground service + MediaSession + audio focus,
  perf-профилирование (ноль аллокаций в цикле — проверить Perfetto'й),
  instrumented smoke, иконка, стор-мелочи.

Порядок M4/M5 можно частично распараллелить, но FX раньше UI: без FFT нет
скоупа, и звук — суть приложения.

## 9. Риски и открытые вопросы

- **Reverb/Limiter parity** — самый вероятный источник «звучит не так».
  Митигция: сравнение WAV-рендеров веб (npm run rec) vs Android
  (офлайн-рендер) на одинаковых пресетах; расхождение — осознанно
  согласовать, как перегенерацию golden.
- **GC/джиттер на Kotlin-аудиопотоке.** Митигация: ноль аллокаций,
  BLOCKING write, буфер 2–4 блока. План Б: Oboe/JNI только для цикла.
- **Хаотические формулы (lorenz/rossler) vs golden** — расхождение
  double-математики накапливается; допуски точечно, свойства вместо
  побитовости.
- Решено с пользователем: отдельный репозиторий; при потере audio focus —
  стоп; share-токены и URL-параметры не переносим; пресеты — JSON-дамп из
  веб-репо; FX-ноды Web Audio воспроизводим по идее/формуле, не 1-в-1;
  планшетный лэйаут НУЖЕН в первой версии (входит в M5).

## 10. Что сознательно НЕ переносим

- iosUnlock, wakelock.ts (заменены платформенными механизмами).
- scripts/smoke|shot|rec.mjs — их роль берут JVM-офлайн-рендер и
  instrumented-тесты.
- Share-токены `#s=…`, URL-параметры (`?preset=`), диплинки — решение
  пользователя: в Android-версии не нужны.
- Worklet-специфику (двойная сборка dev-ESM/prod-IIFE, `?worker&url`).
