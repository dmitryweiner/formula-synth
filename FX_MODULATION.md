# План: модуляция FX (LFO → фильтры/эффекты)

Черновик для обсуждения. Ничего ещё не реализовано — это дизайн-документ.
Продолжение MODULATION.md (фаза 6): распространить матрицу модуляции с
параметров генераторов на параметры эффектов (в первую очередь — cutoff
фильтра, «авто-вау»/свип).

## Идея в одном абзаце

Любой маршрут матрицы модуляции может целиться не только в `формула.параметр`
генератора, но и в параметр модуля эффектов — например `LFO → filter cutoff`,
`LFO → delay time`, `LFO → phaser rate`. Тот же пул LFO, та же матрица, тот же
экран. Разница только в приёмнике: FX-ноды живут в главном потоке (Web Audio),
а не в воркалете.

---

## Ключевые решения (и обоснование)

### 1. Модулятор — на главном потоке, control-rate

FX — это нативные Web Audio-ноды в `engine.ts` (BiquadFilter, DelayNode,
Gain'ы). Их параметры — `AudioParam` или JS-значения, применяемые в
`applyFxParams`. Поэтому FX-модуляцию считаем **на главном потоке** по таймеру
(`requestAnimationFrame` или `setInterval` ~30–50 Гц), а не в воркалете. LFO —
медленные (0.02–8 Гц), для них 30–50 Гц более чем достаточно (сглаживание —
через `setTargetAtTime`). **Никаких изменений воркалета и протокола сообщений
не требуется** — вся FX-модуляция локальна в `engine.ts`.

Альтернатива — нативные LFO-ноды (`OscillatorNode → GainNode → AudioParam`).
Отклонена: нет S&H (random), неудобен exp-маппинг (нативная модуляция
аддитивна), сложнее синк с LFO генераторов. Control-rate переиспользует всё
готовое чистое ядро.

### 2. Переиспользуем чистое ядро и `applyFxParams`

- `lfoValue(lfo, t)` и `effectiveParam(base, l, depth, range, exp)` из
  `src/dsp/mod.ts` работают как есть (S&H и exp-октавы уже поддержаны).
- Применение эффективных значений — через существующий **`applyFxParams`**:
  модулятор собирает «эффективный `FxState`» (база, поверх которой наложены
  промодулированные поля) и зовёт `applyFxParams(effFx)`. Внутренние гварды
  (`lastReverbDecay`, `phaserStagesConnected`) уже не дают дорогой работы, если
  соответствующие поля не менялись. Значит **маппинг «поле → нода» писать не
  надо** — он уже есть.

### 3. `modT` из `ctx.currentTime`

Главнопоточный `modT = ctx.currentTime − startTime`. Приблизительно совпадает с
`modT` генераторов (тоже от sample-clock). Идеальный сэмпл-синк между
control-rate FX и block-rate генераторами не нужен — на слух не важно.

### 4. Курируемый allowlist модулируемых FX-параметров

Не всё модулируется гладко. Разрешаем только «дружелюбные» поля; дискретные и
дорогие — исключаем (см. таблицу ниже).

### 5. Разделение base / live

У генераторов overwrite-then-restore самодостаточен внутри `fill()`. На главном
потоке **база** FX-параметра = значение слайдера; модулятор пишет поверх live.
Правило: ручная правка слайдера обновляет базу; снятие маршрута / стоп —
восстанавливает базу. Это главный новый источник состояния и багов.

## Что переиспользуется (уже оплачено модуляцией генераторов)

- `lfoValue` (5 форм, S&H), `effectiveParam` (кламп/биполяр/exp) — без изменений.
- Типы `ModState` / `LfoDef` / `ModRoute` — с небольшим расширением цели.
- UI-каркас `modmatrix.ts` (пул из 3 LFO, строки маршрутов, add/remove).
- `applyFxParams(fx)` в `engine.ts` — как аппликатор эффективных значений.
- Терпимый разбор `mod` в `schema.ts`.

Новое — только «проброс на главный поток + диапазоны FX + UI-группа».

## Модель данных

Маршрут получает различимую цель: генератор или FX.

```ts
// вариант A: зарезервированный id формулы
interface ModRoute {
  src: number;
  formula: FormulaId | 'fx'; // 'fx' → param это ключ FxState
  param: string;             // ключ слайдера генератора ИЛИ поля FxState
  depth: number;
  exp?: boolean;
}
```

`'fx'` — сентинел; при нём `param` — ключ `FxState` (`filterFreq`, `delayTime`,
`phaserRate`, …). Consumer'ы ветвятся по `formula === 'fx'`. Формат состояния
остаётся аддитивным (v3, `mod?` уже есть), ломать ничего не нужно.

## Диапазоны FX-параметров + allowlist

Сейчас диапазоны FX живут только в `index.html` (min/max слайдеров) — их надо
вынести в данные, напр. `FX_PARAM_RANGES: Record<string, [number, number]>`
(рядом с engine или в новом модуле). Заодно это и есть allowlist: что в таблице
— то модулируемо.

| Модуль | Модулируемо (AudioParam / гладко) | НЕ модулируемо |
|---|---|---|
| Filter | filterFreq, filterQ, filterGain, filterVowel, filterCombFb | filterType (дискретно), режимные скачки |
| Chorus/Flanger | chorusRate, chorusDepth, chorusMix, chorusFb | chorusMode (дискретно) |
| Reverb | reverbMix | reverbDecay (пересчёт импульса — дорого) |
| Delay | delayTime¹, delayFb, delayMix | — |
| Phaser | phaserRate, phaserDepth, phaserFb, phaserMix | phaserStages (пересборка графа) |
| Limiter | limiterThr, limiterRel (нишево) | — |

¹ модуляция `delayTime` даёт питч-артефакты (это по сути фленджер) — оставить,
но пометить.

Частотоподобные поля (filterFreq, chorusRate, phaserRate, delayTime) — **exp по
умолчанию** (детект по имени/списку, как для генераторов).

## FX-модулятор (в `engine.ts`)

```ts
// псевдокод
class AudioEngine {
  private baseFx: FxState;              // текущая база (значения слайдеров)
  private fxRoutes: ModRoute[] = [];    // маршруты с formula==='fx'
  private fxModTimer: number | null = null;

  setBaseFx(fx: FxState) { this.baseFx = fx; if (!this.fxRoutes.length) this.applyFxParams(fx); }

  private tickFxMod() {
    const t = this.ctx.currentTime - this.startTime;
    const eff: FxState = { ...this.baseFx };
    for (const r of this.fxRoutes) {
      const lfo = this.modLfos[r.src];
      const range = FX_PARAM_RANGES[r.param];
      if (!lfo || !range) continue;
      eff[r.param] = effectiveParam(this.baseFx[r.param], lfoValue(lfo, t), r.depth, range, r.exp ?? false);
    }
    this.applyFxParams(eff);       // гварды не дают дорогой работы
  }
  // start/stop таймера: при наличии fxRoutes и running; иначе восстановить baseFx
}
```

Нюансы:
- Таймер стартует, когда есть `fxRoutes` и движок играет; гаснет иначе (тогда
  один раз применяем базу).
- Возможно гасить таймер на `visibilitychange` (как rAF скоупа) — FX-модуляция
  «замрёт», аудио продолжится. Приемлемо.
- Для гладкости можно перевести применяемые поля с `setValueAtTime` на
  `setTargetAtTime` (иначе на быстрых LFO — зиппер). Решить при реализации.

## UI (`modmatrix.ts`)

- В дропдаун приёмника маршрута добавить группу **«Effects»** (сентинел `'fx'`)
  рядом с включёнными генераторами (через `<optgroup>`, как в меню Effects
  preset).
- При выборе `Effects` список параметров = allowlist FX-полей (человекочитаемые
  подписи: «Filter cutoff», «Delay time», «Phaser rate», …).
- exp-чекбокс по умолчанию для частотных FX-полей.
- Индикатор ∿ на модулируемых FX-слайдерах в панели эффектов (как на слайдерах
  генераторов) — опционально, приятно.

## Фазы и ориентировочный объём

1. **Минимальный заход (S)** — «LFO → filter cutoff (+Q)». Модель цели `'fx'`,
   `FX_PARAM_RANGES` на 1–2 поля, FX-модулятор с base/live, группа «Effects» в
   матрице для этих полей. Закрывает главный запрос («вау»/свип). Риск/объём —
   как недавние мелкие правки.
2. **Полный allowlist (M)** — остальные модулируемые FX-поля из таблицы,
   человекочитаемые подписи, exp-дефолты, индикатор ∿ на FX-слайдерах.
3. **Пресеты (S)** — 1–2 демо с FX-модуляцией (авто-вау фильтр, качающийся
   дилей).
4. **Опционально (M+)** — генератор как источник модуляции; drag-to-assign
   (общее с MODULATION.md, фаза 6).

Итог: **средне по объёму** для полной версии (~одна фокус-сессия), **мало** для
минимального «LFO → cutoff». Фундамент (LFO-движок, матрица, exp/S&H,
`applyFxParams`) уже оплачен.

## Подводные камни

- **Не всё модулируется гладко** — держать allowlist; `filterType`,
  `chorusMode`, `phaserStages`, `reverbDecay` — вне его (дискретно / пересборка /
  дорогой пересчёт импульса).
- **base vs live** — единственная существенно новая стейт-логика: правка
  слайдера обновляет базу, снятие маршрута / стоп — восстанавливает.
- **`delayTime` под LFO** — питч-глайд/фленджер-артефакты (это фича, но
  предупредить).
- **Синк с генераторами** — control-rate vs block-rate, приблизительный; на слух
  не важно.
- **Двойной LFO** на chorus/phaser — у них уже есть нативный LFO; модуляция их
  `rate` — «LFO поверх LFO», работает, но слоёно.
- **Троттлинг главного потока** — при занятом UI FX-модуляция может дёргаться;
  для медленных LFO обычно незаметно.
