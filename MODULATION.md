# План: матрица модуляции (mod matrix)

**Статус: фазы 1–5 реализованы** (ядро, схема v3, проброс в движок, UI,
демо-пресеты). Фаза 6 (модуляция FX, drag-to-assign, генератор как источник)
осознанно отложена — см. бэклог в CLAUDE.md. Решения по открытым вопросам
зафиксированы внизу документа.

Цель: превратить статичные числа-параметры в живые, медленно меняющиеся —
пресеты начинают эволюционировать, проект усиливается в своей генеративной
природе.

## Идея в одном абзаце

Любой слайдер генератора вместо фиксированного числа может «качаться»
источником модуляции (LFO / случайный дрейф) с заданной глубиной. «Матрица» —
это список маршрутов «источник → формула.параметр → глубина». На что похоже:
mod matrix в Serum/Vital, патч-панель модульного синта, drag-LFO-onto-knob
в Ableton.

---

## Ключевые решения (и их обоснование)

### 1. Только параметры генераторов, FX — потом

Параметры генераторов живут внутри воркалета; параметры эффектов — это
Web Audio `AudioParam` в `engine.ts`. Это два разных мира модуляции.
**v1 модулирует только параметры генераторов** — там главный эффект и всё
юнит-тестируемо. Модуляция FX (нативными Web Audio LFO-нодами) — отдельная
поздняя фаза.

### 2. Расчёт на control-rate, LFO как чистая функция времени

Параметры не аудио-частотные — модуляцию пересчитываем **раз в блок**
(128 семплов, ~3 мс), а не посемплово. LFO вычисляется как чистая функция
абсолютного времени: `lfo(t) = shape(2π·rate·t + phase)`. Каждый
воркалет-генератор считает LFO **локально**; поскольку все ноды стартуют
одновременно и идут от одного sample-clock, они остаются синхронны **без
обмена сообщениями между потоками**. Это в стиле существующего чистого ядра.

- Нужен отдельный `modT`, который **не сбрасывается** в `reset()` (reset
  перезапускает звук, а не модуляцию).
- `modT` продвигается на `out.length / sr` в каждом `fill()`.

### 3. Модуляция «поверх» базы: overwrite-then-restore

`fill()` читает `p.xxx` по всему большому `switch`. Чтобы не переписывать
все чтения: в начале блока сохранить базовые значения модулируемых
параметров, записать в `p[param]` эффективные значения, отработать блок,
затем восстановить базу. Минимальная правка ядра, `switch` не трогаем.

```ts
// псевдокод внутри fill()
const saved = routes.map(r => this.p[r.param]);
for (const r of routes) {
  const l = lfoValue(lfos[r.src], this.modT);
  this.p[r.param] = applyDepth(saved[i], l, r, ranges[r.param]);
}
// ... основной посемпловый цикл использует this.p как обычно ...
for (let i = 0; i < routes.length; i++) this.p[routes[i].param] = saved[i];
this.modT += n / sr;
```

### 4. Глубина — биполярная доля диапазона

```ts
eff = clamp(base + depth * (max - min) * lfo, min, max)   // lfo ∈ [-1, 1]
```

Диапазоны `[min, max]` лежат в UI-схеме (`src/formulas.ts`), ядро их не знает
— передаём маленькую мапу `param → [min, max]` в генератор вместе с
маршрутами. Для частотных параметров — опциональный **экспоненциальный**
маппинг (модуляция в октавах), см. подводные камни.

## Модель данных

```ts
type LfoShape = 'sine' | 'triangle' | 'saw' | 'square' | 'random'; // random = S&H

interface LfoDef {
  shape: LfoShape;
  rate: number;   // Гц (медленные: ~0.02–8)
  phase: number;  // 0–1
}

interface ModRoute {
  src: number;        // индекс LFO в пуле
  formula: FormulaId; // приёмник — какой генератор
  param: string;      // ключ слайдера
  depth: number;      // биполярно, доля диапазона [-1, 1]
  exp?: boolean;      // экспоненциальный маппинг (для частот)
}

interface ModState {
  lfos: LfoDef[];     // фиксированный небольшой пул
  routes: ModRoute[];
}
```

`lfoValue(lfo, t): number` — чистая функция → `[-1, 1]`. Для `random` (S&H) —
детерминированный хэш от `floor(t · rate)` (чтобы оставаться чистой функцией
времени и синхронной между нодами).

## Схема состояния и совместимость

Разрешено ломать share-ссылки (переехали в другую репу). Поэтому:

- **Бампаем `v: 2 → v: 3`**, добавляем `AppState.mod?: ModState`.
- Изменение всё равно аддитивное, так что новый формат читается штатно; но
  мы **не обязуемся** парсить старые токены и **снимаем** legacy-token тест
  из `serialize.test.ts` как жёсткий контракт (упрощает жизнь).
- `sanitizeState` расширяется терпимым разбором `mod` (неизвестное/битое —
  отбрасываем, как сейчас).
- Ключи localStorage пользовательских пресетов оставить прежними —
  их-то ломать смысла нет (это не про URL).

## Протокол сообщений (воркалет)

Новое сообщение генератору при изменении модуляции:

```ts
{ type: 'mod', lfos: LfoDef[], routes: ModRoute[], ranges: Record<string,[number,number]> }
```

`engine.ts` знает всё; каждой формуле-ноде шлёт **только маршруты,
нацеленные на неё** + полный пул LFO + диапазоны нужных параметров.
Передаём то же и в `processorOptions` при старте (для восстановления из
пресета/URL).

## Тестирование (сохраняем «звук = код, зафиксированный тестами»)

Всё считается в чистом ядре → покрывается в текущем стиле:

- формы LFO по `t` (диапазон, период, детерминизм S&H);
- расчёт эффективного параметра: клампы, биполярность, exp-маппинг;
- две ноды с одинаковыми LFO дают синхронные значения;
- `reset()` НЕ сбивает `modT`;
- round-trip и терпимость `sanitizeState` для блока `mod`;
- golden-семплы промодулированного генератора (новый эталон, сид фиксирован).

## Подводные камни

- **Частота воспринимается логарифмически.** Линейная ±доля-диапазона на `fc`
  (20–2000) вокруг базы 220 качнёт вверх сильнее, чем вниз, — криво. Для
  частотных параметров нужен exp-маппинг (в октавах/полутонах). Заложить
  сразу как флаг `exp` на маршруте, по умолчанию включать для частотных
  слайдеров.
- **Целочисленные параметры** (N гармоник, октавы Shepard, bbRecipe)
  промодулируются в дробные; внутри и так `floor`-ятся — работает ступеньками.
- **`gain`** пересекается с гейтом вкл/выкл (`src/dsp/gate.ts`) — безопасно,
  но помнить: гейт домножает уже посчитанный (возможно промодулированный)
  выход.
- **CPU** — block-rate расчёт дешёвый; выключенные генераторы и так не
  считаются (гейт), модуляция их не касается.

6. Генератор как источник модуляции — **отложено** (фаза 6).

Дополнительно (по ходу реализации):
- Совместимость со старыми share-ссылками **не держим** (переезд в отдельный
  репо), формат состояния — **v3**; legacy-token тест снят.

  маршрут не удаляется.
